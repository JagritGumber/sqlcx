## Usage

```bash
sqlcx check
```

Runs the same pipeline as `sqlcx generate` — loading config, globbing SQL files, partitioning, hashing, and parsing — but **does not write any output files**. Use this to verify your SQL and config are valid before committing or deploying.

---

## Pipeline

`check` and `generate` share the same internal pipeline. The only difference is the final step:

| Step | `generate` | `check` |
|---|---|---|
| Load config | Yes | Yes |
| Glob SQL files | Yes | Yes |
| Partition schema / queries | Yes | Yes |
| Hash & cache check | Yes | Yes |
| Parse SQL | Yes | Yes |
| Write output files | Yes | **No** |

---

## Output

On success, `check` prints a summary to stderr and exits with code `0`:

```
$ sqlcx check
Loading config from /myapp
Scanning SQL files in /myapp/sql
Cache miss — parsing SQL files
Check passed — 3 tables, 8 queries, 1 enums
```

On failure (parse error, bad config, missing file), it prints an error message and exits with a non-zero code:

```
$ sqlcx check
Loading config from /myapp
Scanning SQL files in /myapp/sql
Cache miss — parsing SQL files
error: unknown type "uuid" at sql/schema.sql:12
```

---

## CI/CD usage

Add `sqlcx check` to your CI pipeline to catch SQL or config errors before they reach production.

### GitHub Actions

```yaml
- name: Validate SQL
  run: sqlcx check
```

### Makefile

```makefile
lint:
    sqlcx check
```

`sqlcx check` exits non-zero on any error, so it integrates naturally with `set -e` scripts and CI failure conditions.
