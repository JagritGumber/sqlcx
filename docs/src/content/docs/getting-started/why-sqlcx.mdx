You already know the best language for talking to databases. It's SQL.

The problem isn't SQL — it's everything ORMs build on top of it. Schema DSLs, query builders, runtime engines, type-level gymnastics. Every layer adds weight, complexity, and another thing that can break.

sqlcx takes a different approach: **write SQL, generate code, ship nothing.**

---

## What's wrong with the status quo?

<CardGrid>
  <Card title="Prisma" icon="warning">
    Ships a **1.6 MB** query engine binary. Every Lambda cold start pays for it. Every Docker image carries it. And you can't read the SQL it generates without enabling debug logging.
  </Card>
  <Card title="Drizzle" icon="warning">
    Lighter at ~7.4 KB, but your schema lives in TypeScript, not SQL. Deep generics slow down IDE type-checking on large schemas. If you have a DBA writing migrations in SQL, someone has to translate.
  </Card>
  <Card title="sqlc" icon="warning">
    Got the approach right — SQL-first, zero runtime, typed output. But it only targets Go. TypeScript, Rust, and Python developers are left out.
  </Card>

## The sqlcx approach

Write SQL. Get typed code. Ship nothing.

```
┌─────────────┐     ┌─────────┐     ┌──────────────────┐
│  schema.sql  │────▶│  sqlcx   │────▶│  typed code      │
│  queries.sql │     │  (build) │     │  (TS, Go, Rust,  │
└─────────────┘     └─────────┘     │   Python)         │
                                     └──────────────────┘
                         ↓
                    0 KB runtime
```

- **SQL is the schema.** You write `CREATE TABLE` and named queries. sqlcx reads them.
- **Zero runtime.** Generated code calls your database driver directly. There is nothing between your query and the wire.
- **Every language.** One SQL source, one config — TypeScript, Go, Rust, and Python output.
- **Open validation.** Pick TypeBox, Zod, Pydantic, or Serde. sqlcx plugs into the ecosystem you already use.

> If you know SQL, you already know sqlcx.

## Runtime bundle comparison

| Tool | Runtime bundle | What you're shipping |
|------|---------------|---------------------|
| **sqlcx** | **0 KB** | Nothing. Generated code only. |
| Drizzle | ~7.4 KB | Query builder runtime |
| Prisma | ~1.6 MB | Binary query engine + JS runtime |

sqlcx generates code at build time. At runtime, the only thing running is your SQL hitting the driver. That's it.

## Same query, every language

Write your SQL query once:

```sql
-- name: GetUser :one
SELECT id, name, email, status
FROM users
WHERE id = $1;
```

Use the generated code in any language:

<Tabs>
  <TabItem label="TypeScript">
    ```typescript
    const user = await getUser(sql, { id: 42 });
    console.log(user.name); // fully typed
    ```
  </TabItem>
  <TabItem label="Python">
    ```python
    user = SelectUsers.model_validate(dict(row))
    print(user.name)  # fully typed with Pydantic
    ```
  </TabItem>
  <TabItem label="Go">
    ```go
    user, err := queries.GetUser(ctx, 42)
    fmt.Println(user.Name) // fully typed struct
    ```
  </TabItem>
  <TabItem label="Rust">
    ```rust
    let user = get_user(&pool, 42).await?;
    println!("{}", user.name); // fully typed with serde
    ```
  </TabItem>
</Tabs>

Every version is fully typed from your schema. No hand-written interfaces. No runtime surprises. No `any`.

## Who is sqlcx for?

- **Teams with polyglot backends** — same SQL, different services, different languages, one source of truth
- **Performance-sensitive deployments** — serverless, edge, embedded — where every KB matters
- **SQL-first developers** — you write SQL by hand and want the types to follow, not the other way around
- **Anyone tired of ORMs** — no more fighting abstractions to write the query you actually want
